# 金字塔原理 — 汇报材料逻辑优化器

> 基于 Barbara Minto 《金字塔原理》，提供结构化思维和逻辑表达的完整工具链。
> **核心理念：任何事情都可以归纳出一个中心论点，这个论点可由三至七个论据支持。每个论据又可以成为下一层的论点，如此循环往复，形成金字塔结构。**

---

## 使用方式

### 1. 逻辑重构

```bash
# 分析并重构汇报材料的逻辑结构
/skill comm-pyramid-optimizer restructure <材料路径或文本>

# 按特定框架重构
/skill comm-pyramid-optimizer restructure <材料> --framework <SCQA|总分总|时间序|结构序>
```

**示例**:
```bash
/skill comm-pyramid-optimizer restructure ./quarterly-report.md
/skill comm-pyramid-optimizer restructure "项目进展汇报" --framework SCQA
```

### 2. 逻辑检查

```bash
# 检查材料的逻辑完整性
/skill comm-pyramid-optimizer check <材料路径或文本>

# 检查特定规则
/skill comm-pyramid-optimizer check <材料> --rule <MECE|归纳|演绎|结论先行>
```

**示例**:
```bash
/skill comm-pyramid-optimizer check ./proposal.md
/skill comm-pyramid-optimizer check ./email-draft.md --rule MECE
```

### 3. 结构生成

```bash
# 根据主题和素材生成金字塔结构
/skill comm-pyramid-optimizer generate --topic <主题> --audience <受众> --purpose <目的>

# 生成特定类型的沟通结构
/skill comm-pyramid-optimizer generate --type <汇报|提案|邮件|口头报告>
```

**示例**:
```bash
/skill comm-pyramid-optimizer generate --topic "Q4营收下降分析" --audience "CEO" --purpose "获批改进方案"
/skill comm-pyramid-optimizer generate --type 提案 --topic "引入AI编程助手"
```

### 4. SCQA 构建器

```bash
# 用 SCQA 框架构建开场
/skill comm-pyramid-optimizer scqa --situation <背景> --complication <冲突> --question <问题>
```

**示例**:
```bash
/skill comm-pyramid-optimizer scqa --situation "公司数字化转型已进入第二阶段" --complication "核心系统迁移遇到技术瓶颈" --question "如何在Q2前完成迁移"
```

## 核心原理

### 金字塔结构

```
                ┌──────────┐
                │ 核心结论  │  ← 结论先行
                └────┬─────┘
          ┌──────────┼──────────┐
     ┌────┴────┐┌────┴────┐┌────┴────┐
     │ 论据1   ││ 论据2   ││ 论据3   │  ← MECE 分组
     └────┬────┘└────┬────┘└────┬────┘
    ┌─────┼─────┐ ┌──┼──┐  ┌───┼───┐
    │支撑 │支撑 │ │  │  │  │   │   │  ← 事实/数据支撑
    └─────┴─────┘ └──┴──┘  └───┴───┘
```

### 四大基本原则

| 原则 | 英文 | 含义 | 检查方法 |
|------|------|------|---------|
| **结论先行** | Top-down | 先说结论，再说原因 | 每段/每页的第一句话是否是该段的核心观点？ |
| **以上统下** | Summarize | 上层论点是下层论据的总结 | 删掉下层，上层结论是否成立？不成立则有遗漏 |
| **归类分组** | Group | 同一层的论据属于同一类别 | 同层论据能否用一个名词统称？ |
| **逻辑递进** | Order | 同一组内的论据有逻辑顺序 | 调换顺序是否影响理解？影响则有序 |

## 执行流程

### 逻辑重构流程

```
1. 材料诊断 → 分析现有材料的逻辑结构（或缺乏结构）
2. 核心提取 → 提炼一句话核心结论（So What?）
3. 分组归类 → 将素材按 MECE 原则分组
4. 逻辑排序 → 确定每组内的逻辑顺序
5. 结构搭建 → 构建金字塔结构
6. 检查验证 → 用四大原则逐项检查
7. 优化输出 → 输出重构后的结构大纲
```

### 响应级别

| 级别 | 触发条件 | 输出形式 |
|------|---------|---------|
| **快速** | 简单逻辑检查、概念问答 | 问题指出 + 修改建议 |
| **标准** | 单个材料重构、SCQA 构建 | 结构大纲 + 修改建议 |
| **深度** | 完整汇报材料重构 | 完整金字塔结构 + 逐页建议 |

## 常见逻辑错误与修正

| 错误类型 | 症状 | 修正方法 |
|---------|------|---------|
| **没有结论** | 罗列了很多数据/事实，但不知道要说什么 | 反问"所以呢？(So What?)"，提炼核心结论 |
| **结论在最后** | 铺垫太长，结论在第10页才出现 | 结论前置到第1页，后续作为支撑 |
| **逻辑跳跃** | 论据和结论之间缺少推理链条 | 补充中间推理步骤 |
| **重叠分类** | 同一层的论据有交叉重叠 | 用 MECE 法则重新分类 |
| **遗漏关键** | 分类不完整，缺少重要维度 | 用 MECE 框架（如3C、4P等）检查完整性 |
| **顺序混乱** | 论据排列无序，读者不知道先看哪个 | 按时间序、重要性序或结构序排列 |
| **信息过载** | 每页塞太多信息，主次不分 | 每页一个核心观点，7±2 条目以内 |

## 应用场景

### 场景1：向上汇报

```bash
/skill comm-pyramid-optimizer generate --type 汇报 --topic "Q3项目进展" --audience "VP" --purpose "争取Q4资源"
```

**推荐结构**: SCQA 开场 + 金字塔正文
- S: 项目背景和战略意义
- C: 当前面临的资源瓶颈
- Q: 如何在保证Q4交付的同时扩展能力?
- A: 核心结论（需要增加X人/Y预算）→ 三个支撑论据

### 场景2：方案提案

**推荐结构**: 问题-方案-收益
1. 结论：推荐方案X
2. 论据1：解决了什么问题（现状分析）
3. 论据2：为什么选方案X（方案对比）
4. 论据3：能带来什么收益（ROI分析）

### 场景3：邮件沟通

**推荐结构**: 结论先行 + 行动请求
1. 第一行：核心请求/结论
2. 背景：2-3句话的上下文
3. 理由：为什么需要这样做
4. 行动：明确的行动请求和截止日期

## 强制约束

**必须做**：
1. 永远从受众和目的出发，再选择表达结构
2. 每次重构都要先提炼核心结论
3. 用 MECE 检查分类的完整性和互斥性
4. 给出具体的、可操作的修改建议
5. 用中文回答

**不要做**：
1. 不要只指出问题不给解决方案
2. 不要用教科书语言，要用通俗的表达
3. 不要忽视受众的知识背景和关注点
4. 不要追求完美结构而丢失关键信息
5. 不要对所有场景都用同一个框架
