# 演绎法规则 (Deductive Reasoning)

> **定义**: 从一般性前提推导出具体结论的逻辑推理方法。
> **方向**: 从一般到具体，经典形式为三段论（大前提-小前提-结论）。

---

## 演绎法在金字塔中的作用

```
┌──────────────┐
│   大前提      │  ← 普遍规律/原则
└──────┬───────┘
       ↓
┌──────────────┐
│   小前提      │  ← 具体情况/事实
└──────┬───────┘
       ↓
┌──────────────┐
│   结论        │  ← 由大前提+小前提推出
└──────────────┘
```

---

## 经典三段论结构

### 标准形式

> **大前提**: 所有延迟交付的项目都会导致成本增加
> **小前提**: 我们的项目已经延迟了2个月
> **结论**: 因此，我们的项目成本将会增加

### 商业应用示例

> **大前提**: 系统响应超过3秒会导致30%的用户流失（行业数据）
> **小前提**: 我们的系统平均响应时间已达5秒
> **结论**: 我们正在流失大量用户，必须优化性能

## 演绎法 vs 归纳法

| 维度 | 演绎法 | 归纳法 |
|------|--------|--------|
| **方向** | 一般→具体 | 具体→一般 |
| **确定性** | 结论必然成立（前提正确时） | 结论可能成立 |
| **适用场景** | 有明确规律/原则可依据时 | 需要从数据中发现规律时 |
| **说服力** | 逻辑严密，难以反驳 | 需要足够多的证据 |
| **风险** | 大前提可能错误 | 可能过度概括 |

### 选择建议

- **用演绎法**: 当你有公认的原则、政策或数据可以依据
- **用归纳法**: 当你需要从多个事实中总结出新的结论
- **汇报中推荐**: 用归纳法展示论据，用演绎法构建论证链

## 演绎链（多步演绎）

在复杂论证中，一个演绎的结论可以作为下一个演绎的前提：

```
大前提1: 数字化转型是行业趋势
小前提1: 我们尚未启动数字化
结论1/大前提2: 我们面临竞争力下降的风险
    ↓
小前提2: 竞争对手已经完成了数字化
结论2/大前提3: 我们需要加速数字化转型
    ↓
小前提3: 当前最大瓶颈是核心系统
结论3: 建议优先改造核心系统
```

**注意**: 演绎链不宜超过 3-4 步，否则逻辑链太长，受众容易失去耐心。

---

## 检查清单

- [ ] 大前提是公认的事实、数据或原则（不是主观臆断）
- [ ] 小前提是经过验证的具体事实
- [ ] 结论是由大前提+小前提必然推出的
- [ ] 中间没有逻辑跳跃（每一步推理都成立）
- [ ] 演绎链不超过 3-4 步
- [ ] 大前提有可靠来源（行业数据、政策文件、公认标准等）

---

## 常见错误

| 错误 | 说明 | 修正 |
|------|------|------|
| **大前提错误** | "所有IT项目都会延期"（这不是事实） | 验证大前提的可靠性，换用经过验证的数据 |
| **偷换概念** | 大前提中的概念和小前提中的概念看似相同实则不同 | 确保每一步使用的概念定义一致 |
| **逻辑跳跃** | 从"需要改进"直接跳到"应该全部重写" | 补充中间推理步骤 |
| **因果倒置** | 把相关关系当成因果关系 | 验证因果关系的方向性 |
