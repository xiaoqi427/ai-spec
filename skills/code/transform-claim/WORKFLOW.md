# Transform Claim - AI执行工作流

## 标准执行流程

当用户请求迁移报账单代码时（如 `/skill transform-claim otc T044`），必须按以下流程执行：

---

### 阶段1: 解析请求（立即执行）

**输入解析**:
- 模块名: `otc/ptp/tr/eer/fa/rtr`
- 模板号: `T044/T047/T051` 等
- 步骤范围: 全部(1-10) 或 单步(--step X)
- 需求文档: (可选) --md 路径

**立即输出**:
```
开始迁移: claim-{module} 模块 {模板号}
步骤范围: {1-10 或 步骤X}
需求文档: {有/无}
```

**加载对应步骤文档**:
- 如果是全部步骤: 依次读取 steps/step-01.md 到 step-10.md
- 如果是单步: 只读取 steps/step-0X.md

---

### 阶段2: 分析老代码（必须完成）

**对于每个步骤，必须执行**:

1. **定位老代码文件**
   - 根据步骤文档中的"老代码"映射关系
   - 搜索路径: `yldc-caiwugongxiangpingtai-fsscYR-master/src/com/yili/fssc/claim/service/`
   - 示例: 步骤1 → 搜索 `NewT{XXX}ClaimService.java`

2. **读取并分析老代码**
   - 记录文件总行数
   - 逐行分析 execute() 或主要方法
   - 识别业务逻辑模块

3. **对照步骤文档中的"基类已实现"清单**
   - 标记哪些逻辑基类已有（无需迁移）
   - 标记哪些逻辑需要迁移到子类

**必须输出（标准格式）**:
```
【步骤X: {功能名}】

📄 老代码分析:
- 文件: {路径}
- 总行数: {XX}
- 主方法: execute() / 其他

🔍 逻辑分类:
✅ 基类已实现（无需迁移）:
  - [逻辑点1] (原代码行XX-XX)
  - [逻辑点2] (原代码行XX-XX)
  
🔧 需迁移到子类:
  - [逻辑点1] (原代码行XX-XX) → preExecute()
  - [逻辑点2] (原代码行XX-XX) → 其他方法

📦 依赖的Service/工具:
  - [Service1]
  - [工具类1]
```

---

### 阶段3: 输出设计方案（等待确认）

**设计方案必须包含**:

1. **新代码结构**
   - 接口名: `IT{XXX}{功能}Service`
   - 实现类名: `T{XXX}{功能}ServiceImpl`
   - 继承基类: `Base{功能}Service`

2. **需重写的方法**
   - 方法签名
   - 方法职责
   - 原代码对应行数

3. **需注入的依赖**
   - Service依赖
   - 工具类依赖

4. **预期代码行数**
   - 接口: XX行
   - 实现类: XX行

**输出格式**:
```
【步骤X: 设计方案】

📐 新代码结构:
- 接口: IT{XXX}{功能}Service extends IBase{功能}Service
- 实现类: T{XXX}{功能}ServiceImpl extends Base{功能}Service
- 位置: claim-{module}-service/.../t{xxx}/{层级}/

🔧 重写方法:
1. preExecute(...) 
   - 职责: T{XXX}特有的初始化逻辑
   - 原代码: NewT{XXX}ClaimService.java 第XX-XX行

2. [其他方法]
   - ...

💉 依赖注入:
@Resource
- private I{依赖1}Service {依赖1}Service;
- private {工具类1} {工具类1};

📊 预期代码量:
- 接口: ~15行
- 实现类: ~80行

---
⏸️  请确认设计方案后回复"确认"或"ok"
```

**等待用户确认**: 
- 用户回复"确认"/"ok"/"可以" → 进入阶段4
- 用户提出修改 → 调整方案后重新输出
- 不要自行继续到编码阶段

---

### 阶段4: 编写新代码（确认后执行）

**步骤**:

1. **读取对应模板文件**
   - 接口模板: `templates/{层级}/interface-{功能}-template.java`
   - 实现模板: `templates/{层级}/impl-{功能}-template.java`

2. **替换模板变量**
   - `{{module}}` → 实际模块名(小写)
   - `{{moduleUpper}}` → 首字母大写
   - `{{templateNum}}` → 模板编号
   - `{{templateNumLower}}` → 小写
   - `{{businessDesc}}` → 业务描述
   - `{{author}}` → sevenxiao
   - `{{date}}` → 当前日期

3. **填充业务逻辑**
   - 在对应的钩子方法中填入迁移的老代码逻辑
   - 每段逻辑必须标注原代码行数注释
   - 格式: `// 原代码: NewT{XXX}ClaimService.java 第XX-XX行`

4. **处理依赖**
   - 添加 @Resource 注解的依赖注入
   - 添加必要的 import 语句（文件头部）

5. **创建文件**
   - 使用 `create_file` 工具创建接口和实现类
   - 路径遵循步骤文档中的"新代码存放路径"

**代码质量要求**:
- ✅ 无编译错误
- ✅ 使用 @Slf4j、@Service 注解
- ✅ import 语句完整
- ✅ 不对传入变量重新赋值（对象.set可以）
- ✅ 优先使用 MyBatis Plus 而非手写SQL
- ✅ 不存在的字段创建在DTO中而非DO中

---

### 阶段5: 验证完整性（自动执行）

**对照步骤文档的检查清单验证**:

1. **分析阶段检查**
   - [ ] 老代码文件已定位并记录行数
   - [ ] 老代码逻辑已逐行分析
   - [ ] 基类已实现的逻辑已标记
   - [ ] 需迁移的逻辑已标记

2. **编码阶段检查**
   - [ ] 接口和实现类已创建
   - [ ] 类上有 @Slf4j 和 @Service
   - [ ] 重写方法有完整注释（含原代码行数）
   - [ ] 依赖注入使用 @Resource
   - [ ] import 语句完整

3. **验证阶段检查**
   - [ ] 无编译错误
   - [ ] 老代码每一行都有对应处理
   - [ ] 没有重复基类已实现的逻辑
   - [ ] 没有对传入变量重新赋值

**输出验证结果**:
```
【步骤X: 验证完成】

✅ 已创建文件:
- {接口路径}
- {实现类路径}

✅ 代码行数:
- 接口: XX行
- 实现类: XX行

✅ 检查结果:
- 编译状态: 无错误
- 老代码覆盖: 100% (XX/XX行)
- 基类调用: 正确
- 注释完整性: ✅

📝 迁移总结:
- 基类已实现: [X个逻辑点]
- 子类重写: [X个方法]
- 原代码行数: XX行 → 新代码行数: XX行
```

## 多步骤连续执行

当用户请求迁移全部10个步骤时:

1. **依次执行步骤1-10**
2. **每个步骤完成后**:
   - 输出该步骤的验证结果
   - 自动进入下一步（无需等待确认）
3. **遇到错误时**:
   - 停止执行
   - 输出错误详情
   - 等待用户处理

**进度显示**:
```
【迁移进度: 3/10】
✅ 步骤1: New报账单头 - 已完成
✅ 步骤2: New报账单行 - 已完成
🔄 步骤3: Save报账单头 - 进行中...
⏳ 步骤4-10: 等待中
```

## 特殊情况处理

### 情况1: 老代码文件未找到
```
❌ 错误: 未找到老代码文件
预期路径: {路径}
建议:
1. 检查模板号是否正确
2. 检查老代码库路径
3. 手动指定老代码路径
```

### 情况2: 基类方法不明确
```
⚠️  警告: 基类方法不明确
- 老代码逻辑: {描述}
- 可能的基类方法: [方法1, 方法2]

建议: 请查看基类源码确认，或参考已迁移的T047等模板
```

### 情况3: 需要合并多个老Service
```
📦 合并策略:
- InsertT{XXX}ClaimService (新增逻辑)
- UpdateT{XXX}ClaimService (修改逻辑)
→ 合并到 T{XXX}SaveClaimServiceImpl

分流处理:
- 在 preExecute() 中判断 claimId 是否为空
- 新增逻辑: claimId == null
- 修改逻辑: claimId != null
```

## 常见错误预防

1. **不要在分析阶段就开始写代码** - 必须等用户确认设计方案
2. **不要跳过基类能力对照** - 会导致重复实现或遗漏逻辑
3. **不要忘记标注原代码行数** - 这是强制要求
4. **不要修改模板变量外的模板内容** - 保持模板结构
5. **不要创建编译错误的代码** - 必须确保import完整、类型正确
