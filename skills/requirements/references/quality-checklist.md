# 需求质量检查清单

> 需求文档质量保障的检查清单，涵盖SERU各阶段交付物的质量标准。

---

## 1. 主题域划分检查清单

### 必须通过项

- [ ] **SA-01** 每个主题域有唯一编号和名称
- [ ] **SA-02** 每个主题域有一句话的业务职责描述（不超过30字）
- [ ] **SA-03** 职责单一：一个主题域只承担一类核心业务职责
- [ ] **SA-04** 边界清晰：两个主题域之间没有职责模糊地带
- [ ] **SA-05** 低耦合：主题域间的依赖关系不超过3个
- [ ] **SA-06** 完整覆盖：所有业务领域都被覆盖，没有"无人认领"的业务
- [ ] **SA-07** 系统边界图存在且完整

### 建议检查项

- [ ] **SA-08** 外部接口已全部识别并标注
- [ ] **SA-09** 划分依据和决策记录已记录
- [ ] **SA-10** 主要干系人已识别

---

## 2. 业务事件检查清单

### 必须通过项

- [ ] **EV-01** 事件命名使用"动词+名词"格式（如：提交报销单）
- [ ] **EV-02** 每个事件有明确的触发条件（不是"当需要时"）
- [ ] **EV-03** 每个事件标注了触发类型（外部/内部/时间）
- [ ] **EV-04** 每个事件有参与角色（不是"用户"）
- [ ] **EV-05** 每个事件归属到具体的主题域
- [ ] **EV-06** 没有重复或高度相似的事件
- [ ] **EV-07** 每个主题域至少有3个业务事件

### 建议检查项

- [ ] **EV-08** 事件标注了优先级和频率
- [ ] **EV-09** 事件之间的依赖关系已识别
- [ ] **EV-10** 报表需求已同步识别

---

## 3. 用例规格检查清单

### 必须通过项

- [ ] **UC-01** 用例有唯一编号和名称
- [ ] **UC-02** 简述说明了参与者的**业务目标**，而非系统功能描述
- [ ] **UC-03** 前置条件已列出且可验证
- [ ] **UC-04** 后置条件分为成功和失败两种情况
- [ ] **UC-05** 主成功场景步骤完整，每步明确"谁做什么"
- [ ] **UC-06** 参与者动作和系统响应交替描述
- [ ] **UC-07** 至少有3个扩展/异常场景
- [ ] **UC-08** 用例关联到了具体的业务事件
- [ ] **UC-09** 没有模糊描述（如"系统处理数据""适当时显示"）

### 建议检查项

- [ ] **UC-10** 业务规则已记录且具体可校验
- [ ] **UC-11** 输入/输出数据已定义
- [ ] **UC-12** 非功能性需求有量化指标
- [ ] **UC-13** 多角色差异场景已分离
- [ ] **UC-14** 关联的用例/报表已标注

---

## 4. 领域模型检查清单

### 必须通过项

- [ ] **DM-01** 核心业务实体已识别（覆盖所有主题域）
- [ ] **DM-02** 实体使用业务语言命名（不是技术表名）
- [ ] **DM-03** 实体间关系有方向和基数标注（1:1, 1:N, N:N）
- [ ] **DM-04** 有生命周期的实体有状态流转图

### 建议检查项

- [ ] **DM-05** 关键属性已定义，含业务含义说明
- [ ] **DM-06** 数据量已估算
- [ ] **DM-07** 跨主题域的数据流已标注

---

## 5. SRS 文档检查清单

### 必须通过项

- [ ] **SRS-01** 文档结构完整（引言→概述→功能需求→数据需求→非功能需求→约束）
- [ ] **SRS-02** 术语表完整，全文术语一致
- [ ] **SRS-03** 功能需求按业务主题域组织（不是按技术模块）
- [ ] **SRS-04** 非功能性需求全部量化（有具体数字，如"< 2秒"）
- [ ] **SRS-05** 需求追溯矩阵存在且完整
- [ ] **SRS-06** 没有魔法数字或未定义的常量

### 建议检查项

- [ ] **SRS-07** 变更记录表存在
- [ ] **SRS-08** 约束条件和假设条件已记录
- [ ] **SRS-09** 接口需求已记录（外部系统 + 用户接口）
- [ ] **SRS-10** 报表需求单独列出

---

## 6. 追溯性检查清单

### 必须通过项

- [ ] **TR-01** 每个用例可追溯到业务事件
- [ ] **TR-02** 每个业务事件可追溯到主题域
- [ ] **TR-03** 每个主题域可追溯到业务目标
- [ ] **TR-04** 没有"孤立需求"（无法追溯到上层目标的需求）

### 建议检查项

- [ ] **TR-05** 每个业务目标都有对应的用例实现
- [ ] **TR-06** 数据需求可追溯到业务实体
- [ ] **TR-07** 非功能需求关联到具体用例或场景

---

## 7. 通用质量属性检查

### 无二义性

- [ ] **QA-01** 没有可多种理解的描述
- [ ] **QA-02** 没有"等等""适当""合理""尽量"等模糊词汇
- [ ] **QA-03** 条件描述是明确的（不是"必要时"）

### 完整性

- [ ] **QA-04** 正常场景和异常场景都覆盖
- [ ] **QA-05** 功能需求、数据需求、非功能需求三方面都覆盖
- [ ] **QA-06** 四条主线（人事物接口）都检查过

### 一致性

- [ ] **QA-07** 同一概念在不同文档中名称一致
- [ ] **QA-08** 数据定义在不同用例中保持一致
- [ ] **QA-09** 权限描述在不同角色间无矛盾

### 可测试性

- [ ] **QA-10** 每个需求都有明确的验收条件
- [ ] **QA-11** 非功能性需求有可测量的量化指标
- [ ] **QA-12** 业务规则可以设计测试用例来验证
