# 需求建模图谱指南

> UML 建模在 SERU 各阶段的应用指南，包含 Mermaid 语法示例。

---

## 概述

需求建模通过可视化方式表达需求，弥补自然语言的模糊性。不同阶段使用不同的模型：

| SERU 阶段 | 建模目标 | 推荐模型 | Mermaid 类型 |
|-----------|---------|---------|-------------|
| 需求定义 | 系统边界和主题域 | 系统上下文图 | `graph TB` |
| 需求定义 | 用例概览 | 用例图 | `graph TB` |
| 需求捕获 | 业务流程 | 泳道流程图 | `graph LR` + subgraph |
| 需求捕获 | 数据关系 | 领域类图 / ER图 | `classDiagram` |
| 需求分析 | 对象生命周期 | 状态机图 | `stateDiagram-v2` |
| 需求分析 | 系统交互 | 时序图 | `sequenceDiagram` |

---

## 1. 系统上下文图

### 适用场景
- 项目启动阶段，明确系统边界
- 识别外部参与者和外部系统

### Mermaid 示例

```mermaid
graph TB
    User[申请人] --> System[报销系统]
    Approver[审批人] --> System
    Finance[财务人员] --> System
    System --> ERP[ERP系统]
    System --> Bank[银行系统]
    System --> Invoice[发票系统]
    Admin[系统管理员] --> System
```

### 建模要点
- 中间是目标系统（一个节点）
- 左侧是人（角色），右侧是外部系统
- 箭头表示交互方向
- 不画系统内部细节

---

## 2. 主题域关系图

### 适用场景
- 主题域划分后，展示域间关系

### Mermaid 示例

```mermaid
graph TB
    subgraph 系统边界
        SA01[SA-01: 报销申请管理]
        SA02[SA-02: 审批流程管理]
        SA03[SA-03: 财务核算管理]
        SA04[SA-04: 资金支付管理]
        SA05[SA-05: 基础配置管理]
    end
    
    SA01 --> SA02
    SA02 --> SA03
    SA03 --> SA04
    SA05 --> SA01
    SA05 --> SA02
    SA05 --> SA03
```

### 建模要点
- 使用 subgraph 包裹系统边界
- 箭头表示数据/业务依赖方向
- 不加颜色和样式

---

## 3. 用例图

### 适用场景
- 展示系统功能概览和角色关系

### Mermaid 示例

```mermaid
graph TB
    subgraph 报销申请管理
        UC01[UC-01: 创建报销单]
        UC02[UC-02: 提交报销单]
        UC03[UC-03: 查看报销单]
    end
    subgraph 审批流程管理
        UC04[UC-04: 审批报销单]
        UC05[UC-05: 退回报销单]
    end
    
    申请人 --> UC01
    申请人 --> UC02
    申请人 --> UC03
    审批人 --> UC04
    审批人 --> UC05
```

### 建模要点
- 按主题域分组
- 角色在图的左侧
- 用例用矩形节点表示（Mermaid 不支持椭圆）

---

## 4. 泳道流程图（业务流程图）

### 适用场景
- 跨角色的业务事件流程
- 展示业务流转和决策分支

### Mermaid 示例

```mermaid
graph LR
    subgraph 申请人
        A[填写报销单] --> B[附加发票]
        B --> C[提交申请]
    end
    subgraph 部门经理
        C --> D{审批}
        D -->|通过| E[审批通过]
        D -->|驳回| F[退回修改]
    end
    subgraph 财务部
        E --> G[财务审核]
        G --> H{复核}
        H -->|通过| I[生成付款单]
        H -->|退回| J[退回]
    end
    subgraph 出纳
        I --> K[执行付款]
        K --> L[完成]
    end
    F --> A
    J --> A
```

### 建模要点
- 每个 subgraph 代表一个角色/泳道
- 菱形节点表示决策分支
- 注意异常流程的回路

---

## 5. 领域类图

### 适用场景
- 核心业务实体和关系建模
- 数据需求分析

### Mermaid 示例

```mermaid
classDiagram
    class 报销单 {
        +String 报销单号
        +Date 申请日期
        +BigDecimal 总金额
        +String 状态
        +submit() 提交
        +approve() 审批
    }
    class 报销明细 {
        +String 费用类型
        +BigDecimal 金额
        +String 说明
    }
    class 发票 {
        +String 发票号码
        +BigDecimal 金额
        +Date 开票日期
    }
    class 员工 {
        +String 工号
        +String 姓名
        +String 部门
    }
    
    员工 "1" --> "N" 报销单 : 提交
    报销单 "1" --> "N" 报销明细 : 包含
    报销明细 "1" --> "0..N" 发票 : 关联
```

### 建模要点
- 属性使用业务名称，不用技术字段名
- 标注关系基数（1, N, 0..N, 0..1）
- 方法只列核心业务操作

## 6. 状态机图

### 适用场景
- 单据/对象的生命周期
- 状态流转和触发条件

### Mermaid 示例

```mermaid
stateDiagram-v2
    [*] --> 草稿 : 创建
    草稿 --> 待审批 : 提交
    草稿 --> 已删除 : 删除
    待审批 --> 已通过 : 审批通过
    待审批 --> 已驳回 : 审批驳回
    已驳回 --> 草稿 : 修改重提
    已驳回 --> 已撤销 : 撤销
    已通过 --> 待付款 : 财务确认
    待付款 --> 已完成 : 付款完成
    已完成 --> [*]
    已删除 --> [*]
    已撤销 --> [*]
```

### 建模要点
- 用 `[*]` 表示起始和结束
- 每个转换标注触发条件
- 覆盖所有合法的状态转换路径

---

## 7. 时序图

### 适用场景
- 系统间交互
- 复杂用例的内部流程

### Mermaid 示例

```mermaid
sequenceDiagram
    participant U as 申请人
    participant S as 报销系统
    participant I as 发票验真服务
    participant B as BPM审批引擎
    
    U->>S: 提交报销单
    S->>I: 验证发票真伪
    I-->>S: 返回验证结果
    alt 发票验证通过
        S->>B: 发起审批流程
        B-->>S: 审批流程已创建
        S-->>U: 提交成功，等待审批
    else 发票验证失败
        S-->>U: 提交失败，发票无效
    end
```

### 建模要点
- participant 声明参与者
- `->>` 表示同步调用，`-->>` 表示返回
- 使用 `alt/else/end` 表示分支

---

## 建模原则

1. **业务视角优先**：模型服务于业务理解，不是技术实现
2. **适度抽象**：不要画得太细，保持在需求层面
3. **一致性**：同一实体在不同图中保持名称一致
4. **无样式**：使用基本 Mermaid 语法，不加颜色和 CSS
5. **配合文字**：模型配合文字说明，不能只有图没有解释
