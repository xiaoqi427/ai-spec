# SERU 过程框架详解

> 基于《软件需求最佳实践》（于向东等著）的 SERU 过程框架完整知识体系。

---

## 1. SERU 框架概述

SERU 是一种**业务驱动**的需求分析过程框架，核心思想是：

**需求分析 = 分解 + 抽象**

```
原始需求（零散/矛盾/模糊）
    ↓ 分解（自顶向下）
将目标系统 → 主题域 → 业务事件 → 用例
    ↓ 抽象（自底向上）
消除矛盾（重复/冲突/逻辑不完整）
    ↓
清晰完整的需求体系
```

### 1.1 四要素定义

| 要素 | 英文 | 定义 | 核心作用 |
|------|------|------|---------|
| **S** (Subject Area) | 主题域 | 按业务职责划分的系统子域 | 降低系统复杂度，实现高内聚低耦合 |
| **E** (Event) | 业务事件 | 触发系统响应的业务动作 | 识别系统需要处理的所有业务场景 |
| **R** (Report) | 报表 | 系统对外输出的数据展示 | 完善数据输出需求，避免遗漏 |
| **U** (Use Case) | 用例 | 参与者与系统交互的完整场景 | 详细描述人机交互细节 |

### 1.2 三个阶段

| 阶段 | 目标 | 核心活动 | 交付物 |
|------|------|---------|--------|
| **需求定义** | 明确目标和范围 | 主题域划分(S)、业务事件识别(E)、报表识别(R)、系统边界确定 | 主题域划分文档、业务事件列表、系统上下文图 |
| **需求捕获** | 理清脉络和框架 | 细化事件和报表、UML建模（领域类图+泳道图+用例图）、全局领域模型 | 领域模型、业务流程图、用例图 |
| **需求分析** | 填充需求细节 | 用例详述(U)、事件流描述、前/后置条件、非功能性需求、角色场景分离 | 用例规格、SRS文档 |

## 2. 需求的三个层次

### 2.1 层次关系

```
┌─────────────────────────────────────────────┐
│ 业务需求 (Business Requirements)              │
│ → 组织为什么要建设这个系统                       │
│ → 来源: 项目发起人、高管                         │
│ → 产出: 项目愿景、业务目标、成功指标               │
├─────────────────────────────────────────────┤
│ 用户需求 (User Requirements)                  │
│ → 用户需要完成什么任务/解决什么问题               │
│ → 来源: 最终用户、业务专家                       │
│ → 产出: 用例、用户故事、业务流程                  │
├─────────────────────────────────────────────┤
│ 软件功能需求 (Software Requirements)          │
│ → 系统具体需要实现什么功能                       │
│ → 来源: 需求分析师的分析结果                     │
│ → 产出: SRS文档、功能规格                       │
└─────────────────────────────────────────────┘
```

### 2.2 转化职责

需求工程师的核心职责就是**转化**：

```
客户的自然语言诉求 → 结构化的软件需求
模糊的业务目标     → 可测量的功能规格
零散的用户期望     → 完整的需求体系
```

## 3. 分解方法详解

### 3.1 自顶向下分解

```
目标系统（整体范围）
  ├── 主题域1（业务职责A）
  │   ├── 业务事件1.1
  │   │   ├── 用例1.1.1
  │   │   └── 用例1.1.2
  │   ├── 业务事件1.2
  │   └── 报表1.1
  ├── 主题域2（业务职责B）
  │   ├── 业务事件2.1
  │   └── 业务事件2.2
  └── 主题域3（业务职责C）
```

### 3.2 分解原则

| 原则 | 说明 | 检验标准 |
|------|------|---------|
| **MECE原则** | 相互独立、完全穷尽 | 无遗漏、无重叠 |
| **高内聚** | 同一主题域内的事件高度相关 | 域内事件共享相同的数据实体 |
| **低耦合** | 不同主题域间依赖最小化 | 域间通过明确接口交互 |
| **粒度适当** | 每个用例完成一个完整的业务目标 | 用例不宜太粗（拆分）也不宜太细（合并） |

## 4. 抽象方法详解

### 4.1 自底向上抽象

从具体的用户诉求中，通过抽象提炼共性：

```
具体诉求A: "我要提交差旅报销"
具体诉求B: "我要提交日常报销"        ──抽象──→ 报销申请管理（主题域）
具体诉求C: "我要提交采购报销"

具体诉求D: "经理审批报销单"
具体诉求E: "财务审核报销单"          ──抽象──→ 报销审批管理（主题域）
具体诉求F: "总监终审报销单"
```

### 4.2 消除矛盾

| 矛盾类型 | 表现 | 处理方法 |
|---------|------|---------|
| **重复** | 不同人描述了同一个需求 | 合并去重，保留最完整的版本 |
| **冲突** | 不同干系人的需求相互矛盾 | 追根溯源，找到业务目标层面的优先级 |
| **模糊** | 需求描述不够具体 | 通过追问和场景化具体化 |
| **遗漏** | 某些场景没有被覆盖 | 通过四条主线检查（人事物接口） |

## 5. 与其他方法论的对比

| 维度 | SERU | 传统需求分析 | DDD（领域驱动设计） |
|------|------|------------|------------------|
| **驱动方式** | 业务事件驱动 | 功能列表驱动 | 领域模型驱动 |
| **组织方式** | 主题域 | 功能模块 | 限界上下文 |
| **核心制品** | 用例规格 | 需求列表 | 领域模型 |
| **适用阶段** | 需求分析 | 需求收集 | 设计阶段 |
| **优势** | 业务完整性好 | 简单直接 | 技术实现指导性强 |
| **互补关系** | SERU 的主题域 ≈ DDD 的限界上下文，可以衔接使用 |

## 6. SERU 实践要点

### 6.1 常见误区

| 误区 | 正确做法 |
|------|---------|
| 把"方案"当"需求" | 需求是"要做什么"，方案是"怎么做" |
| 只关注功能需求 | 必须同时关注数据需求和非功能需求 |
| 从技术视角划分主题域 | 从业务职责视角划分 |
| 跳过业务事件直接写用例 | 先识别事件，确保不遗漏 |
| 用例粒度不当 | 每个用例对应一个完整的业务目标 |

### 6.2 成功标准

- 每个业务目标都能追溯到具体的用例实现
- 每个用例都能追溯到业务目标
- 没有"孤立"的需求（无法追溯到上层目标）
- 需求描述无二义性，开发和测试都能理解
- 非功能性需求全部量化
