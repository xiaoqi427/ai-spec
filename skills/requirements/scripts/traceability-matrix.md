# 需求追溯矩阵生成脚本

> 自动生成需求追溯矩阵，确保从业务目标到用例的完整追溯链路。

---

## 用途

在需求分析完成后，生成需求追溯矩阵，用于：
1. 验证需求覆盖完整性（正向追溯）
2. 验证需求必要性（反向追溯）
3. 变更影响分析的基础数据

---

## 追溯维度

### 正向追溯链路

```
业务目标 (G-XX)
  └→ 主题域 (SA-XX)
      └→ 业务事件 (E-XX)
          └→ 用例 (UC-XX)
              └→ 数据实体 (EN-XX)
                  └→ 非功能需求 (NFR-XX)
```

### 反向追溯链路

```
非功能需求 (NFR-XX)
  └→ 数据实体 (EN-XX)
      └→ 用例 (UC-XX)
          └→ 业务事件 (E-XX)
              └→ 主题域 (SA-XX)
                  └→ 业务目标 (G-XX)
```

---

## 执行流程

### Step 1: 收集需求要素

```
从需求文档中提取以下编号列表:
- 业务目标: G-01, G-02, ...
- 主题域: SA-01, SA-02, ...
- 业务事件: E-01, E-02, ...
- 用例: UC-01, UC-02, ...
- 数据实体: EN-01, EN-02, ...
- 报表: R-01, R-02, ...
- 非功能需求: NFR-P01, NFR-S01, ...
- 外部接口: IF-01, IF-02, ...
```

### Step 2: 建立关联关系

```
从需求文档中提取关联关系:
- 业务目标 → 主题域: 从主题域文档的"系统目标"章节
- 主题域 → 业务事件: 从业务事件列表的"所属主题域"列
- 业务事件 → 用例: 从用例规格的"触发业务事件"字段
- 用例 → 数据实体: 从用例规格的"数据需求"章节
- 用例 → 报表: 从用例规格的"关联"章节
```

### Step 3: 生成追溯矩阵

```
横轴: 需求要素（目标、主题域、事件、用例、实体）
纵轴: 需求要素
交叉点: 关联关系（有/无）
```

### Step 4: 覆盖性分析

```
检查:
1. 每个业务目标是否被至少一个用例覆盖
2. 每个业务事件是否有对应的用例
3. 每个用例是否可追溯到业务目标
4. 是否存在"孤立"的需求（无法追溯到上层目标）
5. 是否存在"悬空"的目标（没有任何用例实现）
```

## 输出格式

### 完整追溯矩阵

```markdown
## 需求追溯矩阵

**项目名称**：[项目名称]
**生成日期**：[YYYY-MM-DD]
**需求版本**：v[X.X]

### 正向追溯：业务目标 → 用例

| 业务目标 | 主题域 | 业务事件 | 用例 | 覆盖状态 |
|---------|--------|---------|------|---------|
| G-01 [目标名] | SA-01 | E-01, E-02 | UC-01, UC-02, UC-03 | ✅ 已覆盖 |
| G-02 [目标名] | SA-02 | E-05 | UC-08 | ✅ 已覆盖 |
| G-03 [目标名] | SA-03 | — | — | ❌ 未覆盖 |

### 反向追溯：用例 → 业务目标

| 用例 | 业务事件 | 主题域 | 业务目标 | 追溯状态 |
|------|---------|--------|---------|---------|
| UC-01 | E-01 | SA-01 | G-01 | ✅ 可追溯 |
| UC-15 | — | SA-04 | — | ⚠️ 孤立需求 |

### 数据追溯：用例 → 数据实体

| 用例 | 输入实体 | 输出实体 | 数据覆盖 |
|------|---------|---------|---------|
| UC-01 | EN-01, EN-02 | EN-01 | ✅ 完整 |
| UC-05 | EN-03 | — | ⚠️ 无输出 |

### 覆盖性报告

| 检查项 | 结果 | 详情 |
|--------|------|------|
| 业务目标覆盖率 | [X/Y = XX%] | 未覆盖: [G-XX] |
| 业务事件覆盖率 | [X/Y = XX%] | 未覆盖: [E-XX] |
| 孤立需求数 | [N 个] | [UC-XX, UC-XX] |
| 悬空目标数 | [N 个] | [G-XX] |

### 问题清单

| 序号 | 问题类型 | 描述 | 建议处理 |
|------|---------|------|---------|
| 1 | 目标未覆盖 | G-XX 没有对应的用例实现 | 补充用例或确认是否在范围内 |
| 2 | 孤立需求 | UC-XX 无法追溯到业务目标 | 确认必要性或建立追溯关系 |
| 3 | 事件遗漏 | E-XX 没有对应的用例 | 补充用例 |
```

---

## 注意事项

1. 追溯矩阵应在每次需求变更后更新
2. 覆盖率低于 90% 时需要重点关注未覆盖的目标
3. 孤立需求需要确认是否真的需要，还是遗漏了追溯关系
4. 追溯矩阵是变更影响分析的基础数据
