# SERU 分析引擎

> 从项目代码或文档中自动提取 SERU 四要素（主题域、业务事件、报表、用例），生成初始需求框架。

---

## 用途

当用户提供了项目代码路径或需求文档时，本脚本指导 AI 系统性地提取 SERU 要素，避免遗漏。

---

## 执行流程

### 阶段1：信息采集

```
输入: 项目路径 或 需求文档路径
```

**1.1 从代码中提取（如有代码路径）**

```bash
# Step 1: 扫描项目结构，识别模块划分
# 目标: 理解现有的技术模块，推断业务主题域
Glob("**/src/main/java/**/controller/**/*.java")  → 识别 Controller 层入口
Glob("**/src/main/java/**/service/**/*.java")      → 识别 Service 层业务逻辑
Glob("**/src/main/java/**/DO/**/*.java")            → 识别数据实体

# Step 2: 从 Controller 提取 API 端点
# 目标: 每个 API 端点可能对应一个业务事件
Grep("@(Get|Post|Put|Delete)Mapping") → 提取所有 REST 端点

# Step 3: 从实体类提取数据模型
# 目标: 识别核心业务实体和关系
Grep("@Table|@Entity|@TableName") → 提取所有数据库映射实体

# Step 4: 从配置中提取外部接口
# 目标: 识别系统边界
Grep("@FeignClient|RestTemplate|WebClient") → 提取外部系统调用
```

**1.2 从文档中提取（如有需求文档）**

```bash
# 读取需求文档
Read(文档路径)

# 提取关键信息
# - 业务目标和范围描述
# - 角色/干系人列表
# - 功能列表/需求条目
# - 流程描述
# - 数据需求
# - 报表需求
```

### 阶段2：SERU 要素提取

**2.1 主题域识别（S）**

```
提取规则:
1. 从代码包结构中识别业务模块（如 claim/, config/, fund/）
2. 从需求文档中提取业务领域关键词
3. 按"高内聚低耦合"原则组织为主题域
4. 验证: 每个主题域有明确的业务职责边界

输出: 主题域列表 SA-01, SA-02, ...
```

**2.2 业务事件识别（E）**

```
提取规则:
1. 每个 Controller 的 API 端点 → 候选业务事件
2. 从需求文档的动词+名词短语 → 候选业务事件
3. 定时任务/消息监听 → 内部/时间触发事件
4. 合并同一业务流程的多个细粒度操作为一个事件
5. 验证: 每个事件有明确的触发条件和参与角色

输出: 业务事件列表 E-01, E-02, ...
```

**2.3 报表识别（R）**

```
提取规则:
1. 从代码中查找导出/报表相关接口（export, report, download）
2. 从需求文档中提取"查询""统计""报表""导出"相关描述
3. 识别仪表盘/大屏需求
4. 验证: 每个报表有明确的数据来源和使用场景

输出: 报表列表 R-01, R-02, ...
```

**2.4 用例候选（U）**

```
提取规则:
1. 每个业务事件 → 至少一个候选用例
2. CRUD 操作按业务含义合并（不是技术操作粒度）
3. 复杂流程拆分为多个用例（每个用例一个业务目标）
4. 验证: 每个用例对应一个完整的业务目标

输出: 用例候选列表 UC-01, UC-02, ...
```

### 阶段3：框架生成

```
根据提取的 SERU 要素，自动生成:
1. 主题域划分文档（使用 subject-area.md 模板）
2. 业务事件列表（使用 business-event-list.md 模板）
3. 领域模型草案（使用 domain-model.md 模板）
4. 用例清单（用例名 + 所属主题域 + 触发事件）

标注: 自动提取结果标记为"待确认"，需人工审核
```

---

## 注意事项

1. 自动提取结果仅作为初始框架，**必须经过人工确认**
2. 代码分析侧重技术视角，需要补充业务视角的理解
3. 文档分析可能有遗漏，需要追问确认完整性
4. 置信度标注帮助用户优先审核低置信度的内容
