# 需求质量检查脚本

> 自动化检查需求文档的质量，确保需求完整、清晰、可追溯、无二义性。

---

## 用途

在需求文档完成后，使用本脚本进行质量门禁检查，确保交付物满足 SERU 标准。

---

## 检查维度

### 维度1：主题域划分质量

```
检查项:
□ [SA-完整性] 每个主题域有唯一编号 (SA-XX)
□ [SA-职责单一] 每个主题域职责描述不超过一句话
□ [SA-低耦合] 主题域间依赖关系明确且最小化
□ [SA-覆盖完整] 所有业务领域都被覆盖，无遗漏
□ [SA-边界清晰] 系统边界图存在且完整
□ [SA-外部接口] 所有外部系统接口已识别

评分: 通过 (6/6) / 基本通过 (4-5/6) / 不通过 (<4/6)
```

### 维度2：业务事件质量

```
检查项:
□ [EV-命名规范] 所有事件命名为"动词+名词"格式
□ [EV-触发条件] 每个事件有明确的触发条件
□ [EV-触发类型] 每个事件标注了触发类型 (外部/内部/时间)
□ [EV-角色明确] 每个事件有参与角色
□ [EV-归属明确] 每个事件归属到具体主题域
□ [EV-优先级] 每个事件标注了优先级
□ [EV-完整性] 每个主题域至少有3个业务事件
□ [EV-无重复] 没有重复或高度相似的事件

评分: 通过 (8/8) / 基本通过 (6-7/8) / 不通过 (<6/8)
```

### 维度3：用例质量

```
检查项:
□ [UC-编号唯一] 每个用例有唯一编号 (UC-XX)
□ [UC-目标明确] 简述说明了参与者的业务目标
□ [UC-前置条件] 前置条件已列出
□ [UC-后置条件] 成功/失败后置条件已列出
□ [UC-主事件流] 主成功场景步骤完整，每步明确"谁做什么"
□ [UC-扩展场景] 至少有3个扩展/异常场景
□ [UC-业务规则] 相关业务规则已记录
□ [UC-数据需求] 输入/输出数据已定义
□ [UC-非功能量化] 非功能性需求有量化指标
□ [UC-事件关联] 每个用例关联到业务事件
□ [UC-无模糊] 没有模糊描述（如"系统处理数据"）
□ [UC-角色分离] 多角色差异场景已分离

评分: 通过 (12/12) / 基本通过 (9-11/12) / 不通过 (<9/12)
```

### 维度4：领域模型质量

```
检查项:
□ [DM-实体完整] 所有核心业务实体已识别
□ [DM-属性明确] 关键属性已定义，含业务含义
□ [DM-关系标注] 实体关系有方向和基数 (1:1, 1:N, N:N)
□ [DM-状态流转] 有生命周期的实体有状态图
□ [DM-命名规范] 使用业务语言命名，非技术术语
□ [DM-数据量估算] 核心实体有数据量估算

评分: 通过 (6/6) / 基本通过 (4-5/6) / 不通过 (<4/6)
```

### 维度5：SRS 文档质量

```
检查项:
□ [SRS-结构完整] 包含引言、概述、功能需求、数据需求、非功能需求等必备章节
□ [SRS-术语定义] 术语表完整，全文用语一致
□ [SRS-按主题域组织] 功能需求按业务主题域组织
□ [SRS-追溯矩阵] 需求追溯矩阵存在且完整
□ [SRS-NFR量化] 所有非功能性需求有量化指标
□ [SRS-版本记录] 变更记录表存在
□ [SRS-无魔法数字] 没有未定义的常量或数字
□ [SRS-无二义性] 没有可多种理解的描述

评分: 通过 (8/8) / 基本通过 (6-7/8) / 不通过 (<6/8)
```

### 维度6：追溯性

```
检查项:
□ [TR-目标追溯] 每个需求可追溯到业务目标
□ [TR-事件追溯] 每个用例可追溯到业务事件
□ [TR-数据追溯] 数据需求可追溯到业务实体
□ [TR-无孤立需求] 没有无法追溯到业务目标的"孤立需求"
□ [TR-正向覆盖] 每个业务目标都有对应的用例实现
□ [TR-反向验证] 每个用例都有验证其正确性的方式

评分: 通过 (6/6) / 基本通过 (4-5/6) / 不通过 (<4/6)
```

---

## 执行策略

```
1. 读取待检查的需求文档
2. 逐维度执行检查项
3. 记录每个检查项的结果（通过/不通过/不适用）
4. 对不通过项给出具体问题描述和修改建议
5. 生成质量检查报告
```

---

```markdown

**检查人员**：[AI / 姓名]

|------|-----------|------|------|

| 用例 | X/12 | [评分] | [状态] |

| SRS文档 | X/8 | [评分] | [状态] |
| 追溯性 | X/6 | [评分] | [状态] |

2. **建议改进**：[列出一般问题]

```

---

## 注意事项

1. 检查项中"不适用"的项目不计入总分
2. 所有"严重"级别问题必须修复后才能通过质量门禁
3. 质量检查结果应作为需求评审的输入
