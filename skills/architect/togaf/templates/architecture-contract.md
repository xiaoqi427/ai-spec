## 4. 架构规范遵从要求

### 4.1 分层架构

```
Controller → Business → DO Service → Mapper → Database
```

- ✅ Controller 只调用 Business 层
- ✅ Business 可调用多个 DO Service
- ✅ DO Service 只调用 Mapper
- ❌ 禁止跨层调用
- ❌ 禁止反向调用
- ❌ DO Service 禁止 if/SQL/业务逻辑

### 4.2 代码规范

- 遵守《阿里巴巴 Java 开发手册》
- 代码格式: Spotless + Google Java Format
- 静态分析: SonarLint + Alibaba Java Guidelines
- 单元测试覆盖率: Service 层 > 80%

### 4.3 API 规范

- RESTful API 设计
- OpenAPI 3.0 文档
- 统一返回结构 `Result<T>`
- 版本化管理

### 4.4 安全规范

- 身份认证和授权
- 数据加密（传输/存储）
- SQL 注入防护
- 敏感信息保护
- 审计日志

## 7. 偏差处理

### 7.1 偏差定义

| 偏差等级 | 定义 | 处理方式 | 时限 |
|----------|------|---------|------|
| 轻微 | 不影响功能和安全的规范偏差 | 记录，下迭代修复 | 1 迭代 |
| 一般 | 不符合架构标准但不影响核心功能 | 限期整改 | 2周 |
| 严重 | 违反架构原则或安全要求 | 暂停发布，立即整改 | 即时 |

### 7.2 偏差豁免

特殊情况可申请偏差豁免：
1. 提交豁免申请，说明原因和影响
2. 域架构师评审
3. 必要时提交 ARB 审批
4. 记录豁免决策和条件

## 8. 违约条款

| 违约类型 | 触发条件 | 后果 |
|----------|---------|------|
| 一般违约 | 偏差未在规定时间内修复 | 升级为严重偏差 |
| 严重违约 | 持续违反架构原则 | 暂停项目，报告管理层 |
| 重大违约 | 造成生产事故或安全事件 | 启动应急响应 |
