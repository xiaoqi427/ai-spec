# 架构治理框架

> 架构治理确保架构实施符合组织的架构原则、标准和决策。
> 对应 TOGAF 10 ADM 阶段 G（实施治理）和阶段 H（变更管理）。

---

## 1. 治理组织结构

```
ARB（架构评审委员会）
├── 业务架构治理组
├── 信息系统架构治理组（数据+应用）
└── 技术架构治理组

支撑角色:
├── 首席架构师 ─ 总体架构方向和决策
├── 域架构师 ─ 各域架构设计和评审
├── 解决方案架构师 ─ 项目级架构设计
└── 架构管理员 ─ 架构存储库管理
```

### 1.1 ARB（架构评审委员会）职责

| 职责 | 描述 |
|------|------|
| 架构决策审批 | 审批重大架构决策（ADR） |
| 标准制定 | 制定和维护架构原则和技术标准 |
| 合规评审 | 主持重大项目的架构合规评审 |
| 变更审批 | 审批重大架构变更 |
| 争议仲裁 | 解决架构设计争议 |
| 资产管理 | 监督架构存储库的维护 |

### 1.2 角色与职责矩阵（RACI）

| 活动 | 首席架构师 | 域架构师 | 解决方案架构师 | 项目经理 | ARB |
|------|-----------|---------|---------------|---------|-----|
| 架构原则定义 | A | R | C | I | A |
| 架构愿景制定 | A | R | R | C | I |
| 域架构设计 | C | A/R | R | I | I |
| 合规性评审 | A | R | R | I | A |
| 变更审批（重大） | R | C | I | I | A |
| 变更审批（一般） | A | R | I | I | I |

**RACI**: R=执行 A=负责 C=咨询 I=知情

## 2. 治理流程

### 2.1 核心治理流程

```
架构需求 → 域组评审 → ARB决议 → 实施监督 → 合规验证 → 归档
```

### 2.2 详细流程

#### 流程1: 架构评审流程

```
1. 项目申请架构评审
   ├── 提交架构设计文档
   ├── 提交合规性自检清单
   └── 指定评审范围和时间

2. 域架构组预审
   ├── 检查架构原则遵从
   ├── 检查技术标准遵从
   ├── 识别偏差和风险
   └── 出具预审意见

3. ARB 正式评审（如需要）
   ├── 听取项目汇报
   ├── 审议域组预审意见
   ├── 讨论偏差处置
   └── 作出决议

4. 决议执行
   ├── 合规 → 放行
   ├── 有条件合规 → 限期整改，复审
   └── 不合规 → 驳回，重新设计
```

#### 流程2: 变更管理流程

```
1. 变更请求提交
   ├── 填写变更申请表
   ├── 描述变更原因和范围
   └── 初步影响评估

2. 变更分类
   ├── 紧急 → 24h 快速通道
   ├── 重大 → ARB 审批（2周）
   ├── 一般 → 域组审批（1周）
   └── 标准 → 自动批准

3. 影响分析
   ├── 四域影响评估
   ├── 利益相关者分析
   ├── 风险评估
   └── 实施方案制定

4. 审批决策
   ├── 批准 → 进入实施
   ├── 有条件批准 → 补充条件后实施
   └── 拒绝 → 记录原因，关闭

5. 实施监督
   └── 合规验证 → 归档
```

## 3. 变更分类标准

| 分类 | 审批流程 | 时限 | 审批权限 | 典型场景 |
|------|---------|------|---------|---------|
| **紧急** | 快速通道，事后补审 | 24小时 | 首席架构师 | 生产事故、安全漏洞 |
| **重大** | ARB 全体审批 | 2周 | ARB | 架构原则变更、新技术引入、重大重构 |
| **一般** | 域组审批 | 1周 | 域架构师 | 服务接口调整、中间件配置变更 |
| **标准** | 自动批准 | 即时 | 自动 | 文档更新、小型 Bug 修复 |

### 分类决策树

```
这个变更会影响架构原则或标准吗？
├── 是 → 重大变更
└── 否 → 是生产紧急问题吗？
          ├── 是 → 紧急变更
          └── 否 → 会影响跨服务接口或数据模型吗？
                    ├── 是 → 一般变更
                    └── 否 → 标准变更
```

## 4. 合规性检查框架

### 4.1 检查维度

| 维度 | 检查内容 | 频率 |
|------|---------|------|
| 架构原则 | 是否遵从已定义的架构原则 | 每次评审 |
| 技术标准 | 是否使用批准的技术栈 | 每次评审 |
| 分层架构 | 是否遵守分层调用规则 | 每次评审 |
| 安全合规 | 认证授权、数据加密、审计日志 | 每次评审 |
| 性能合规 | 是否满足非功能需求 | 里程碑评审 |
| 数据合规 | 数据治理、数据质量 | 定期评审 |
| 接口合规 | API 设计规范、兼容性 | 每次评审 |

### 4.2 自动化检查点

| 检查项 | 检查方式 | 工具 |
|--------|---------|------|
| 代码规范 | 静态分析 | SonarLint + Alibaba Guidelines |
| 代码格式 | 自动格式化 | Spotless + Google Java Format |
| 依赖版本 | 依赖扫描 | Maven Enforcer |
| 安全漏洞 | 依赖审计 | OWASP Dependency Check |
| API 规范 | 接口检查 | OpenAPI Validator |

## 5. 架构合同

### 5.1 架构合同内容

架构合同是项目实施方与架构方之间的正式约定。

```
架构合同内容:
1. 约束范围 ─ 适用的系统和模块
2. 架构原则 ─ 必须遵从的原则列表
3. 技术标准 ─ 必须使用的技术栈
4. 质量标准 ─ 代码质量、测试覆盖率要求
5. 评审计划 ─ 合规评审的时间和方式
6. 偏差处理 ─ 发现偏差时的处理流程
7. 签署方 ─ 项目负责人和架构负责人
```

### 5.2 违约处理

| 违约等级 | 触发条件 | 处理方式 |
|----------|---------|---------|
| 轻微 | 非功能性偏差，影响可控 | 记录，下次迭代修复 |
| 一般 | 功能性偏差，不影响核心功能 | 限期整改（2周内） |
| 严重 | 违反架构原则或安全要求 | 暂停发布，立即整改 |
| 紧急 | 生产安全风险 | 立即回滚，启动应急 |

## 6. 治理度量

### 6.1 关键度量指标

| 指标 | 定义 | 目标值 | 度量频率 |
|------|------|--------|----------|
| 架构合规率 | 通过合规评审的项目比例 | > 90% | 月度 |
| 偏差关闭率 | 已关闭偏差 / 总偏差 | > 80% | 月度 |
| 变更响应时间 | 从提交到审批的平均时间 | < 5天 | 月度 |
| 标准遵从率 | 代码扫描通过率 | > 95% | 双周 |
| 架构评审覆盖率 | 评审项目 / 总项目 | 100% | 季度 |
| 架构资产复用率 | 复用 ABB/SBB 的项目比例 | > 50% | 季度 |

### 6.2 治理成熟度模型

| 级别 | 名称 | 特征 |
|------|------|------|
| 1 | 初始 | 无正式治理流程，临时性决策 |
| 2 | 管理 | 有基本流程，但执行不一致 |
| 3 | 定义 | 标准化流程，一致性执行 |
| 4 | 度量 | 量化管理，持续跟踪指标 |
| 5 | 优化 | 持续改进，基于数据决策 |

## 7. 实践建议

1. **从轻量开始** — 不要一开始就建立重量级治理体系
2. **关注价值** — 治理的目的是支撑业务，不是制造官僚
3. **自动化优先** — 能自动检查的不要人工检查
4. **透明公开** — 治理规则和决策对所有人可见
5. **持续改进** — 定期回顾治理流程的有效性
