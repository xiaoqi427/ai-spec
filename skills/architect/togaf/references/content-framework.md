# 架构内容框架

> TOGAF 架构内容框架定义了架构工作的三类产出物：交付物、制品和构建块。
> 这是 TOGAF 10 Fundamental Content 的核心组成部分。

---

## 1. 内容框架概览

```
架构内容框架
├── 交付物（Deliverables）─ 正式文档，需利益相关者批准
├── 制品（Artifacts）    ─ 目录、矩阵、图表
└── 构建块（Building Blocks）
    ├── ABB（Architecture BB）─ 需要什么能力
    └── SBB（Solution BB）   ─ 如何实现
```

---

## 2. 交付物（Deliverables）

交付物是经过正式评审和批准的架构文档，通常面向利益相关者。

| ADM 阶段 | 交付物 | 描述 |
|----------|--------|------|

| C | 数据架构定义文档 | 数据架构的详细设计 |
| C | 应用架构定义文档 | 应用架构的详细设计 |
| D | 技术架构定义文档 | 技术架构的详细设计 |
| E | 架构定义文档（合并） | 四域架构的完整定义 |

## 3. 制品（Artifacts）

制品是描述架构的具体工作产品，分为三类：目录、矩阵、图表。

### 3.1 目录（Catalogs）

目录是某一类架构元素的清单。

| 目录名称 | ADM 阶段 | 描述 |
|----------|----------|------|
| 架构原则目录 | 预备 | 所有架构原则 |
| 业务能力目录 | B | 业务能力清单 |
| 组织/角色目录 | B | 组织和角色清单 |
| 业务服务目录 | B | 业务服务清单 |
| 数据实体目录 | C | 数据实体清单 |
| 应用组合目录 | C | 应用/服务清单 |
| 技术标准目录 | D | 批准使用的技术 |
| 构建块目录 | B-D | ABB 和 SBB 清单 |

### 3.2 矩阵（Matrices）

矩阵表示架构元素间的关系。

| 矩阵名称 | ADM 阶段 | 描述 |
|----------|----------|------|
| 利益相关者矩阵 | A | 利益相关者与关注点 |
| 业务交互矩阵 | B | 组织间交互关系 |
| 角色/功能矩阵 | B | 角色与业务功能的映射 |
| 数据实体/业务功能矩阵 | C | 数据与业务的关系 |
| 应用/功能矩阵 | C | 应用与业务功能映射 |
| 应用/技术矩阵 | D | 应用与技术栈映射 |

## 4. 构建块（Building Blocks）

### 4.1 ABB（Architecture Building Block）

ABB 定义了需要的能力，是**技术无关**的。

**特征**:
- 描述"做什么"而非"怎么做"
- 关注功能和非功能需求
- 定义接口规范
- 可在多个项目中复用

**ABB 组成要素**:

```
ABB
├── 能力描述 ─ 提供什么能力
├── 功能需求 ─ 需要实现的功能列表
├── 非功能需求 ─ 性能/可用性/安全/可扩展
├── 接口规范
│   ├── 提供的接口 ─ 对外暴露的能力
│   └── 依赖的接口 ─ 需要的外部能力
└── 约束条件 ─ 技术、组织、标准约束
```

### 4.2 SBB（Solution Building Block）

SBB 是 ABB 的具体实现方案，是**技术相关**的。

**特征**:
- 描述"怎么做"
- 包含具体的技术选型
- 定义集成方式和运维要求
- 与 ABB 一一对应或多对一

**SBB 组成要素**:

```
SBB
├── 技术选型 ─ 使用什么技术/产品
├── 实现方案 ─ 具体的实现设计
├── 集成方式 ─ 与其他系统的对接
├── 运维要求 ─ 监控/扩缩容/备份/SLA
└── 成本估算 ─ 许可证/资源/人力
```

### 4.3 ABB → SBB 映射示例

```
ABB-APP-001: 统一认证能力
  ├── SBB-APP-001a: OAuth2 + Spring Security 方案
  └── SBB-APP-001b: CAS 单点登录方案

ABB-DATA-001: 数据缓存能力
  ├── SBB-DATA-001a: Redis Cluster 方案
  └── SBB-DATA-001b: Caffeine 本地缓存方案

ABB-TECH-001: 消息异步处理能力
  ├── SBB-TECH-001a: RabbitMQ 方案
  └── SBB-TECH-001b: Kafka 方案
```

## 5. 架构内容元模型

TOGAF 定义了架构内容的元模型，描述了各元素之间的关系：

```
业务能力 ──使用──→ 业务服务 ──被实现──→ 应用组件
    │                                        │
    ├─被──→ 组织单元                         ├─使用──→ 技术组件
    │                                        │
    └─产生──→ 数据实体 ←──管理──┘            └─遵从──→ 技术标准
```

### 核心元素关系

| 源元素 | 关系 | 目标元素 |
|--------|------|----------|
| 业务服务 | 由...实现 | 应用组件 |
| 应用组件 | 部署在 | 技术组件 |
| 数据实体 | 被...管理 | 应用组件 |
| 业务能力 | 包含 | 业务服务 |
| 组织单元 | 拥有 | 业务能力 |
| 应用组件 | 遵从 | 技术标准 |

## 6. 使用建议

1. **按需选择制品** — 不需要产出所有制品，根据项目规模选择
2. **保持一致性** — 不同制品之间的元素命名要一致
3. **渐进细化** — 从高层图表开始，逐步细化到目录和矩阵
4. **版本管理** — 所有交付物和制品都需要版本管理
5. **受众适配** — 不同制品面向不同受众，注意表达方式
