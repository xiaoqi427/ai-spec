# ADM 架构开发方法 — 阶段详解

> ADM（Architecture Development Method）是 TOGAF 的核心方法论，
> 定义了从架构愿景到实施治理的完整开发流程。TOGAF 10 强调 ADM 的迭代性和适应性。

---

## 2. 预备阶段（Preliminary）

### 目的
建立组织的架构能力，为 ADM 周期做好准备。

### 关键活动
1. **定义架构组织** — 确定架构团队、角色和职责
2. **识别架构框架** — 选择和定制 TOGAF 框架
3. **定义架构原则** — 建立指导架构决策的基本原则
4. **选择工具** — 确定架构工具和建模语言
5. **定制 ADM** — 根据组织特点裁剪 ADM 流程

### 关键交付物
| 交付物 | 描述 |
|--------|------|
| 架构原则目录 | 所有架构原则的正式定义 |
| 定制化的架构框架 | 适应组织的 TOGAF 框架 |
| 架构团队章程 | 团队组织、角色、职责 |
| 治理框架定义 | 架构治理的基本流程 |

### 原则定义模板

| 要素 | 说明 |
|------|------|
| **名称** | 原则的简短名称 |
| **声明** | 一句话描述原则的核心主张 |
| **理由** | 为什么需要这个原则 |
| **影响** | 遵循和违反此原则的后果 |

### 常见架构原则示例
- **技术多样性最小化** — 减少技术栈数量，降低运维成本
- **数据是资产** — 数据被当作企业资产管理
- **互操作性** — 系统间应能够互相通信和协作
- **安全优先** — 安全要求在设计阶段就要考虑

## 3. 阶段 A：架构愿景（Architecture Vision）

### 目的
定义架构工作的范围、约束和期望，获得利益相关者的共识和批准。

### 关键活动
1. **识别利益相关者** — 映射所有相关方及其关注点
2. **确定业务目标** — 明确架构工作的业务驱动力
3. **定义架构范围** — 确定覆盖的业务域和系统边界
4. **创建架构愿景** — 描述目标架构的高层视图
5. **获得批准** — 取得利益相关者对架构工作说明书的批准

### 关键交付物
| 交付物 | 描述 |
|--------|------|
| 架构工作说明书 | 定义范围、方法、时间表 |
| 架构愿景文档 | 目标架构的高层描述 |
| 利益相关者矩阵 | 利益相关者及其关注点 |
| 业务场景清单 | 关键业务场景和需求 |

### 利益相关者分析矩阵

| 利益相关者 | 角色 | 关注点 | 影响力 | 参与级别 |
|-----------|------|--------|--------|---------|
| [高管] | 赞助者 | 投资回报、战略对齐 | 高 | 知情+决策 |
| [业务部门] | 使用者 | 功能满足、易用性 | 中 | 参与+验证 |
| [IT团队] | 实施者 | 可行性、工作量 | 中 | 深度参与 |
| [运维团队] | 运营者 | 可运维性、可监控 | 低 | 咨询+验证 |

---

## 4. 阶段 B：业务架构（Business Architecture）

### 目的
开发业务架构的基线和目标状态，识别差距。

### 关键活动
1. **业务能力建模** — 识别和分层组织业务能力
2. **业务流程建模** — 描述关键业务流程
3. **组织建模** — 定义组织结构和角色
4. **差距分析** — 对比基线和目标的差距

### 关键制品
| 制品类型 | 制品名称 | 描述 |
|----------|---------|------|
| 目录 | 业务能力目录 | 所有业务能力的层次化清单 |
| 目录 | 组织/角色目录 | 组织单元和角色清单 |
| 矩阵 | 业务交互矩阵 | 组织间/流程间交互关系 |
| 图表 | 业务流程图 | 关键业务流程可视化 |
| 图表 | 业务能力地图 | 能力分层可视化 |

---

## 5. 阶段 C：信息系统架构（Information Systems Architecture）

### 目的
开发数据架构和应用架构。

### C1 数据架构

**关键活动**:
1. 概念数据模型 → 逻辑数据模型
2. 数据实体/属性定义
3. 数据流与生命周期分析
4. 数据治理规则定义

**关键制品**:
| 制品 | 描述 |
|------|------|
| 数据实体目录 | 所有数据实体及属性 |
| 数据生命周期图 | 数据从创建到归档的全流程 |
| 数据流图 | 系统间数据流转关系 |
| 数据治理矩阵 | 数据质量、安全、合规规则 |

### C2 应用架构

**关键活动**:
1. 应用组合分析（现有应用清单）
2. 应用功能/服务定义
3. 应用通信关系分析
4. TIME 评估（容忍/投资/迁移/淘汰）

**关键制品**:
| 制品 | 描述 |
|------|------|
| 应用组合目录 | 所有应用/服务清单 |
| 应用通信图 | 应用间集成关系 |
| TIME 评估矩阵 | 应用处置决策 |
| 应用/功能矩阵 | 应用与业务功能的对应 |

---

## 6. 阶段 D：技术架构（Technology Architecture）

### 目的
开发支撑信息系统架构的技术基础设施架构。

### 关键活动
1. 定义技术标准目录
2. 技术参考模型（TRM）定制
3. 基础设施设计（计算、存储、网络）
4. 中间件和平台选型
5. 安全架构设计

### 关键制品
| 制品 | 描述 |
|------|------|
| 技术标准目录 | 批准使用的技术清单 |
| 技术参考模型 | 分层技术组件参考 |
| 平台分解图 | 技术平台组成 |
| 网络/处理图 | 网络拓扑和部署结构 |
| 非功能需求矩阵 | 性能/可用性/安全要求 |

---

## 7. 阶段 E：机会与解决方案（Opportunities & Solutions）

### 目的
识别实施方案，将差距转化为工作包。

### 关键活动
1. 将差距映射到工作包
2. 评估工作包依赖关系
3. 制定初始迁移计划
4. 识别复用和采购机会

### 关键交付物
| 交付物 | 描述 |
|--------|------|
| 工作包清单 | 可实施的工作单元 |
| 初始迁移计划 | 工作包的初步排列 |
| 过渡架构定义 | 从基线到目标的中间态 |

---

## 8. 阶段 F：迁移规划（Migration Planning）

### 目的
制定详细的迁移和实施计划。

### 关键活动
1. 确定工作包优先级
2. 制定详细迁移计划
3. 生成架构路线图
4. 资源和预算规划

### 关键交付物
| 交付物 | 描述 |
|--------|------|
| 架构路线图 | 时间线上的工作包安排 |
| 详细迁移计划 | 每个工作包的实施计划 |
| 实施与迁移计划 | 资源、预算、时间表 |

---

## 9. 阶段 G：实施治理（Implementation Governance）

### 目的
确保实施项目符合架构定义。

### 关键活动
1. 架构合规性评审
2. 架构合同签署
3. 实施监督和指导
4. 偏差管理

### 关键交付物
| 交付物 | 描述 |
|--------|------|
| 架构合同 | 实施方与架构方的约定 |
| 合规性评审报告 | 定期评审结果 |
| 偏差处置记录 | 偏差及处置方式 |

---

## 10. 阶段 H：架构变更管理（Architecture Change Management）

### 目的
管理架构的持续演进和变更请求。

### 关键活动
1. 建立变更管理流程
2. 评估变更请求
3. 影响分析
4. 决定是否启动新的 ADM 周期

### 变更驱动分类
| 驱动类型 | 示例 | 响应 |
|----------|------|------|
| 技术变更 | 新框架发布、安全补丁 | 评估技术标准更新 |
| 业务变更 | 新业务需求、合规要求 | 可能触发新 ADM 周期 |
| 环境变更 | 云迁移、组织重组 | 评估架构影响范围 |

---

## 11. 需求管理（Requirements Management）

### 目的
贯穿 ADM 全过程的需求管理。

### 核心职责
1. **需求获取** — 从每个 ADM 阶段收集需求
2. **需求分类** — 区分业务需求和架构需求
3. **需求优先级** — 基于业务价值和紧迫性排列
4. **需求追踪** — 确保需求在架构中得到体现
5. **需求变更** — 管理需求变更对架构的影响

---

## 12. ADM 迭代模式

TOGAF 10 强调 ADM 不是线性瀑布流程，支持以下迭代模式：

| 迭代模式 | 描述 | 适用场景 |
|----------|------|----------|
| **架构开发迭代** | 在 B-D 阶段间迭代 | 架构定义阶段 |
| **过渡规划迭代** | 在 E-F 阶段间迭代 | 迁移规划阶段 |
| **架构治理迭代** | 在 G-H 阶段间迭代 | 实施和运营阶段 |
| **ADM 周期迭代** | 完整 ADM 周期重复 | 重大变更驱动 |

**关键原则**: ADM 的每个阶段都可以根据实际情况裁剪或迭代，不必严格顺序执行。
