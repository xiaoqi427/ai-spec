# 四大架构域速查手册

> TOGAF 定义了四大架构域：业务架构、数据架构、应用架构、技术架构。
> 本文档提供每个域的核心概念、关键制品和决策要点速查。

---

## 1. 业务架构（Business Architecture）

### ADM 阶段: B

### 核心概念

业务架构定义了组织的业务战略、治理、组织结构和关键业务流程。

### 关键活动

```
业务能力地图 → 流程建模 → 组织/角色矩阵 → 业务服务目录
```

### 关键制品

| 制品类型 | 名称 | 描述 | 用途 |
|----------|------|------|------|
| 目录 | 业务能力目录 | 所有业务能力的层次化清单 | 识别组织能力全景 |
| 目录 | 组织/角色目录 | 组织单元和角色清单 | 明确职责分工 |
| 目录 | 业务服务目录 | 业务对外提供的服务 | 识别服务边界 |
| 矩阵 | 业务交互矩阵 | 组织间/流程间交互关系 | 分析协作关系 |
| 矩阵 | 角色/功能矩阵 | 角色与业务功能的映射 | 权限和分工设计 |
| 图表 | 业务能力地图 | 能力分层可视化 | 高管沟通 |
| 图表 | 业务流程图 | 关键业务流程 | 流程分析优化 |

### 能力地图模板

```
Level 1: 战略能力
├── Level 2: 核心能力域
│   ├── Level 3: 具体能力
│   │   └── Level 4: 能力细项
│   └── Level 3: 具体能力
└── Level 2: 核心能力域
```

**示例（财务共享）**:
```
财务共享服务能力
├── 报账管理
│   ├── 费用报账 ─ 差旅报销、日常费用
│   ├── 资产报账 ─ 固定资产、低值易耗品
│   └── 项目报账 ─ 项目费用、预算控制
├── 审批管理
│   ├── 审批流程 ─ 流程编排、节点配置
│   └── 审批规则 ─ 金额控制、权限校验
├── 资金管理
│   ├── 付款管理 ─ 付款指令、银企直连
│   └── 对账管理 ─ 银行对账、差异处理
└── 凭证管理
    ├── 凭证生成 ─ 自动生成、规则映射
    └── 凭证过账 ─ ERP同步、科目映射
```

### 决策要点

- 业务能力是否完整覆盖业务需求？
- 业务流程是否存在冗余或断点？
- 组织职责是否清晰，有无重叠？
- 业务服务边界是否合理？

## 2. 数据架构（Data Architecture）

### ADM 阶段: C（前半段）

### 核心概念

数据架构定义了组织的数据资产结构、数据管理策略和数据治理规则。

### 关键活动

```
概念/逻辑数据模型 → 数据治理矩阵 → 数据流图 → 数据生命周期
```

### 关键制品

| 制品类型 | 名称 | 描述 | 用途 |
|----------|------|------|------|
| 目录 | 数据实体目录 | 所有数据实体及属性 | 统一数据定义 |
| 矩阵 | 数据实体/业务功能矩阵 | 数据与业务的关系 | 分析数据归属 |
| 矩阵 | 数据治理矩阵 | 数据质量、安全、合规规则 | 数据治理落地 |
| 图表 | 概念数据模型 | 核心数据实体关系 | 高层数据设计 |
| 图表 | 逻辑数据模型 | 详细数据结构 | 指导物理实现 |
| 图表 | 数据流图 | 系统间数据流转 | 集成设计 |
| 图表 | 数据生命周期图 | 数据全生命周期 | 数据管理策略 |

### 数据治理矩阵模板

| 数据实体 | 数据所有者 | 分类 | 安全级别 | 质量规则 | 保留策略 |
|----------|-----------|------|---------|---------|---------|
| [实体名] | [部门/角色] | 主数据/交易数据/参考数据 | 公开/内部/机密 | [规则] | [保留年限] |

### 数据生命周期

```
创建 → 存储 → 使用 → 共享 → 归档 → 销毁
  │       │       │       │       │       │
  ├─质量校验 ├─加密存储 ├─访问控制 ├─脱敏处理 ├─压缩归档 └─安全删除
  └─格式验证 └─备份策略 └─审计日志 └─权限管理 └─冷热分离
```

### 决策要点

- 数据模型是否覆盖所有业务实体？
- 数据治理规则是否明确？
- 跨系统数据一致性如何保证？
- 敏感数据保护措施是否到位？

## 3. 应用架构（Application Architecture）

### ADM 阶段: C（后半段）

### 核心概念

应用架构定义了组织的应用组合、应用间交互方式和应用处置策略。

### 关键活动

```
应用组合目录 → 应用通信图 → TIME评估 → 安全矩阵
```

### 关键制品

| 制品类型 | 名称 | 描述 | 用途 |
|----------|------|------|------|
| 目录 | 应用组合目录 | 所有应用/服务清单 | 应用全景图 |
| 矩阵 | 应用/功能矩阵 | 应用与业务功能映射 | 分析功能覆盖 |
| 矩阵 | TIME 评估矩阵 | 应用处置策略 | 指导投资决策 |
| 矩阵 | 应用/技术矩阵 | 应用与技术栈映射 | 技术统一性分析 |
| 图表 | 应用通信图 | 应用间集成关系 | 集成设计 |
| 图表 | 应用交互图 | 用户与应用的交互 | 用户体验设计 |

### TIME 评估矩阵

| 分类 | 含义 | 策略 | 投资方向 |
|------|------|------|---------|
| **T**olerate（容忍） | 现状可接受，暂不投资 | 维持现状，最小维护 | 低 |
| **I**nvest（投资） | 战略重要，持续投入 | 增强功能，提升性能 | 高 |
| **M**igrate（迁移） | 功能有价值但技术落后 | 迁移到新平台 | 中高 |
| **E**liminate（淘汰） | 不再有价值 | 计划下线 | 无（只有退出成本） |

**示例**:
| 应用 | TIME | 理由 | 行动计划 |
|------|------|------|---------|
| 老报账系统 | M | 功能完善但技术老旧 | 迁移到新微服务架构 |
| fssc-claim-service | I | 核心业务，持续迭代 | 持续增强 |
| 手工Excel报表 | E | 已被系统化报表替代 | 下线 |
| fssc-config-service | T | 稳定运行 | 最小维护 |

### 决策要点

- 应用组合是否有冗余或功能重叠？
- 应用间集成方式是否合理（同步/异步）？
- 哪些应用该投资、迁移或淘汰？
- 新应用是否符合架构原则？

## 4. 技术架构（Technology Architecture）

### ADM 阶段: D

### 核心概念

技术架构定义了支撑信息系统的技术基础设施、中间件和技术标准。

### 关键活动

```
技术标准目录 → TRM(技术参考模型) → 网络/处理图 → 非功能需求矩阵
```

### 关键制品

| 制品类型 | 名称 | 描述 | 用途 |
|----------|------|------|------|
| 目录 | 技术标准目录 | 批准使用的技术清单 | 技术选型约束 |
| 图表 | 技术参考模型 | 分层技术组件参考 | 技术全景图 |
| 图表 | 平台分解图 | 技术平台组成 | 平台设计 |
| 图表 | 网络/处理图 | 网络拓扑和部署结构 | 基础设施设计 |
| 矩阵 | 非功能需求矩阵 | 性能/可用性/安全要求 | 质量保证 |

### 技术参考模型（TRM）分层

```
┌─────────────────────────────────────────┐
│              应用层                       │
│  Spring Boot, MyBatis Plus, MapStruct    │
├─────────────────────────────────────────┤
│              中间件层                     │
│  Redis, RabbitMQ, Nacos, Sentinel        │
├─────────────────────────────────────────┤
│              数据层                       │
│  MySQL, MongoDB, Elasticsearch            │
├─────────────────────────────────────────┤
│              平台层                       │
│  Spring Cloud, Docker, Kubernetes         │
├─────────────────────────────────────────┤
│              基础设施层                   │
│  Linux, 云服务器, 负载均衡, CDN           │
├─────────────────────────────────────────┤
│              网络层                       │
│  TCP/IP, HTTP/HTTPS, VPN                 │
└─────────────────────────────────────────┘
```

### 非功能需求矩阵模板

| 服务 | 响应时间 | TPS | 可用性 | 数据量 | 安全级别 |
|------|---------|-----|--------|--------|---------|
| [服务名] | < [x]ms | [x] | [99.x]% | [x]GB/年 | 高/中/低 |

### 决策要点

- 技术栈是否统一，有无不必要的多样性？
- 基础设施是否满足非功能需求？
- 安全架构是否覆盖所有层面？
- 技术债务如何管理？

## 5. 四域协同分析

### 跨域关系

```
业务架构 ──────→ 驱动 ──────→ 数据架构
    │                           │
    │                           ▼
    └──────→ 驱动 ──────→ 应用架构
                                │
                                ▼
                           技术架构
```

### 跨域一致性检查

| 检查项 | 说明 |
|--------|------|
| 业务→数据 | 每个业务实体是否有对应的数据实体？ |
| 业务→应用 | 每个业务能力是否有对应的应用支撑？ |
| 数据→应用 | 每个数据实体是否被某个应用管理？ |
| 应用→技术 | 每个应用是否使用批准的技术栈？ |
| 业务→技术 | 业务的非功能需求是否被技术架构满足？ |
