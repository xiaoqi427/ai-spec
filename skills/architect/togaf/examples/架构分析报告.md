## 1. 背景

### 1.1 业务驱动

**伊利财务共享平台 (yldc-caiwugongxiangpingtai-fsscYR)** 是基于 IBM GBS 组件构建的企业级财务共享服务中心，服务于伊利集团及优然牧业，支撑 **200+ 财务人员**、日均 **5000+ 报账单**、年处理 **100万+** 张单据。

当前系统因多年演进面临重大技术债务，需要架构级别的迁移和重构。

### 1.2 范围

本次分析覆盖 `yldc-caiwugongxiangpingtai-fsscYR-master/` 目录下的**全部代码和配置资产**，作为 ADM 阶段 B-D 的现状基线。

### 1.3 关键利益相关者

| 角色 | 关注点 |
|------|--------|
| 业务方(财务共享中心) | 业务连续性、功能完整性 |
| 开发团队 | 迁移工作量、技术选型、代码复杂度 |
| 运维团队 | 部署方式变更、监控体系适配 |
| 管理层 | 迁移周期、风险控制、投资回报 |

## 2. 架构原则

本次分析遵循以下架构原则：

| 编号 | 原则 | 声明 | 对本项目的影响 |
|------|------|------|----------------|
| AP-01 | **业务连续性优先** | 迁移过程不得影响财务业务正常运转 | 必须支持灰度/双跑 |
| AP-02 | **渐进式演进** | 不做大爆炸式重写，采用模块化逐步迁移 | 按业务域逐个迁移 |
| AP-03 | **标准化优先** | 优先采用行业标准和成熟方案 | Spring Boot 3 + MyBatis Plus |
| AP-04 | **解耦与分层** | 严格遵守分层架构，消除跨层依赖 | Controller → Business → DoService → Mapper |
| AP-05 | **可观测性** | 所有组件必须支持监控和追踪 | 日志规范化、链路追踪 |

## 6. 关键架构决策 (ADR)

### ADR-001：单体到微服务的拆分策略

**状态**：已接受

**决策**：按业务领域拆分为 12+ 微服务，每个服务独立部署。

**理由**：
- 老系统 56 个模块高度耦合，52 万行 Java 代码难以维护
- 微服务支持独立开发、部署、扩展
- 新系统已规划了完整的微服务架构

**风险**：分布式事务复杂度增加、服务间调用延迟

### ADR-002：ORM 从 Hibernate 迁移到 MyBatis Plus

**状态**：已接受

**决策**：放弃 Hibernate 3.x + 958 个 hbm.xml，改用 MyBatis Plus。

**理由**：
- Hibernate 3.x 已 EOL，958 个映射文件维护成本极高
- MyBatis Plus 更灵活，单表用 Wrapper，多表用 XML
- 团队熟悉度更高

### ADR-003：前后端分离

**状态**：已接受

**决策**：4202 个 JSP 页面全部重写为 Vue 3 前端。

**理由**：
- JSP + jQuery 1.7 技术栈无法满足现代交互需求
- 前后端分离提升开发效率和用户体验
- 新前端框架 (fssc-web / fssc-web-framework) 已建立

## 11. 治理建议

### 11.1 迁移过程治理

- **架构评审**：每个微服务域迁移前，需进行架构评审，确认分层正确
- **代码质量门禁**：新代码必须通过 Spotless + SonarLint 检查
- **老代码行数标注**：迁移代码需注释老代码对应行号，便于追溯
- **不修改只重写**：迁移原则"不修改代码，需要变更请重写"

### 11.2 合规性检查清单

- [ ] 新代码严格遵守四层架构 (Controller → Business → DoService → Mapper)
- [ ] DoService 层无 if/SQL/业务逻辑
- [ ] 使用 MapStruct 而非反射复制
- [ ] 使用 Lombok 注解
