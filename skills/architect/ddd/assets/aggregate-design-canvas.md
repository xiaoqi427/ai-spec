# 聚合设计画布（Aggregate Design Canvas）

> 来源: Vaughn Vernon 的聚合设计画布，用于系统化地设计聚合根的边界、不变量和协作关系。

---

## 填写指南

### 1. 聚合目的
- 用一两句业务语言描述聚合存在的理由
- 回答"为什么需要这个聚合"而非"它是什么"

### 2. 状态转换
- 列出聚合的核心生命周期状态
- 标注每个状态转换的触发条件

### 3. 不变量/业务规则
- 聚合内部必须始终满足的约束
- 使用自然语言表达
- 每条规则可在代码中对应一个校验方法

### 4. 组成部分
- 聚合根是入口
- 内部实体有独立标识但不能脱离聚合根存在
- 值对象无标识，可替换

### 5-6. 命令与事件
- 命令是输入（用户或系统触发）
- 事件是输出（描述已发生的事实）
- 一个命令可以产生多个事件

### 7. 外部依赖
- 只通过ID引用其他聚合
- 需要的领域服务

### 8. 性能/并发考量
- 影响聚合大小的决策因素
- 并发修改的处理策略

## 示例1: 报账单聚合

```
┌─────────────────────────────────────────────────────────────────┐
│                    聚合名称: Claim（报账单）                       │
│                    所属上下文: 报账上下文                           │
├─────────────────────┬───────────────────────────────────────────┤
│                     │                                           │
│  1. 聚合目的         │  2. 状态转换                               │
│                     │                                           │
│  管理报账单的完整     │  DRAFT → SUBMITTED → APPROVED → PAID      │
│  生命周期，确保       │          ↓                                 │
│  报账金额与行项       │       REJECTED → DRAFT (退回修改)           │
│  始终一致。          │                                           │
│                     │  • submit(): DRAFT → SUBMITTED             │
│                     │  • approve(): SUBMITTED → APPROVED         │
│                     │  • reject(): SUBMITTED → REJECTED          │
│                     │  • revise(): REJECTED → DRAFT              │
│                     │                                           │
├─────────────────────┼───────────────────────────────────────────┤
│                     │                                           │
│  3. 不变量           │  4. 组成部分                               │
│                     │                                           │
│  • 报账总金额 =      │  聚合根: Claim                              │
│    所有行项金额之和   │  内部实体: ClaimLine                        │
│  • 草稿状态才可编辑   │  值对象: Money, ClaimNo, DateRange           │
│  • 至少一行才可提交   │                                            │
│  • 行项金额不能为负   │  聚合大小: ~1个头 + 10-50个行                 │
│                     │                                           │
├─────────────────────┼───────────────────────────────────────────┤
│                     │                                           │
│  5. 处理的命令       │  6. 产生的事件                              │
│                     │                                           │
│  • CreateClaim      │  • ClaimCreatedEvent                       │
│  • AddClaimLine     │  • ClaimLineAddedEvent                     │
│  • RemoveClaimLine  │  • ClaimLineRemovedEvent                   │
│  • SubmitClaim      │  • ClaimSubmittedEvent                     │
│  • ApproveClaim     │  • ClaimApprovedEvent                      │
│  • RejectClaim      │  • ClaimRejectedEvent                      │
│                     │                                           │
├─────────────────────┼───────────────────────────────────────────┤
│                     │                                           │
│  7. 外部依赖         │  8. 性能/并发考量                           │
│                     │                                           │
│  引用的聚合:         │  预估大小: 1头 + ≤50行                       │
│  • UserId(申请人)    │  并发频率: 低（单人操作）                      │
│  • DeptId(部门)      │  一致性: 聚合内强一致                        │
│  • InvoiceId(发票)   │  锁策略: 乐观锁(version字段)                 │
│  调用的服务: 无       │                                           │
│                     │                                           │
└─────────────────────┴───────────────────────────────────────────┘
```

## 示例2: 审批流程聚合

```
┌─────────────────────────────────────────────────────────────────┐
│                    聚合名称: ApprovalProcess（审批流程）             │
│                    所属上下文: 审批上下文                            │
├─────────────────────┬───────────────────────────────────────────┤
│                     │                                           │
│  1. 聚合目的         │  2. 状态转换                               │
│                     │                                           │
│  管理审批流程的       │  CREATED → IN_PROGRESS → COMPLETED        │
│  创建、推进和完结，   │                       → REJECTED           │
│  确保审批节点按       │                                           │
│  顺序执行。          │  • start(): CREATED → IN_PROGRESS          │
│                     │  • approveNode(): 推进到下一节点             │
│                     │  • rejectNode(): → REJECTED                │
│                     │  • complete(): → COMPLETED                 │
│                     │                                           │
├─────────────────────┼───────────────────────────────────────────┤
│                     │                                           │
│  3. 不变量           │  4. 组成部分                               │
│                     │                                           │
│  • 节点必须按顺序     │  聚合根: ApprovalProcess                    │
│    审批              │  内部实体: ApprovalNode                     │
│  • 已完成的节点不     │  值对象: ApprovalDecision, ApproverId       │
│    可再修改          │                                            │
│  • 所有节点通过才     │  聚合大小: 1流程 + 3-10个节点                 │
│    算流程完成        │                                            │
│                     │                                           │
├─────────────────────┼───────────────────────────────────────────┤
│                     │                                           │
│  5. 处理的命令       │  6. 产生的事件                              │
│                     │                                           │
│  • CreateProcess    │  • ProcessCreatedEvent                     │
│  • ApproveNode     │  • NodeApprovedEvent                       │
│  • RejectNode      │  • NodeRejectedEvent                       │
│  • AddCountersign  │  • ProcessCompletedEvent                   │
│                     │  • ProcessRejectedEvent                    │
│                     │                                           │
├─────────────────────┼───────────────────────────────────────────┤
│                     │                                           │
│  7. 外部依赖         │  8. 性能/并发考量                           │
│                     │                                           │
│  引用:              │  预估大小: 1流程 + ≤10节点                    │
│  • sourceId(源单ID)  │  并发频率: 中（多人审批）                     │
│  • UserId(审批人)    │  一致性: 聚合内强一致                        │
│  服务:              │  锁策略: 乐观锁                              │
│  • ApprovalPolicy   │                                           │
│                     │                                           │
└─────────────────────┴───────────────────────────────────────────┘
```

## 检查清单

- [ ] 聚合目的清晰，使用业务语言
- [ ] 状态转换覆盖所有生命周期
- [ ] 不变量列表完整
- [ ] 组成部分遵循"小聚合"原则
- [ ] 命令和事件对应正确
- [ ] 外部依赖仅使用ID引用
- [ ] 性能和并发策略已考虑
