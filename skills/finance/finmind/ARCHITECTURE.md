# FinMind 架构设计：三层递进 + 四智体协同

## 核心设计理念

FinMind 采用**流水线架构**，数据从底层向上流动，经过逐层加工，最终在顶层输出可执行的交易决策。
每一层只关注自己的职责，层与层之间通过**标准化数据格式**通信。

## 二、各层职责定义

### 2.1 数据层（Data Layer）— "眼睛与耳朵"

**核心目标**: 打破信息孤岛，将外部杂乱数据转化为内部标准化格式。

| 技能 | 输入 | 输出 | 数据源 |
|------|------|------|--------|
| 行情获取 Market Data Fetcher | 标的代码 + 周期 | 标准化 OHLCV 数据 | AKShare / 东方财富 / Yahoo Finance |
| 基本面解析 Fundamentals Parser | 标的代码 | 结构化财务指标 | 巨潮资讯 / AKShare / 雪球 |
| 另类数据采集 Alt Data Collector | 关键词 / 标的 | 新闻列表 + 情绪标签 | search_web / 财联社 / 雪球 |

**标准化输出格式**:
```json
{
  "symbol": "600519",
  "name": "贵州茅台",
  "market": "A",
  "data_type": "market|fundamental|alternative",
  "timestamp": "2026-03-15T10:00:00+08:00",
  "data": { ... },
  "source": "akshare|eastmoney|xueqiu",
  "quality_score": 0.95
}
```

### 2.2 研究层（Research Layer）— "大脑"

**核心目标**: 对数据层素材进行深度加工，挖掘逻辑关系，输出量化评分。

| 技能 | 输入 | 输出 | 知识融合 |
|------|------|------|---------|
| 情绪分析 | 新闻/社媒数据 | 情绪评分(-1~+1) | 行为金融学·认知偏差 |
| 因子评分引擎 | 财务指标 + 行情 | 多维因子评分(1-10) | 芒格·段永平·张磊 |
| 事件冲击分析 | 事件描述 + 持仓 | 冲击量化评估 | 达利欧·债务周期 |
| 技术形态识别 | K线数据 | 形态信号 + 关键位 | 利弗莫尔·游子/柚子 |
| 康波周期分析 | 宏观指标 | 周期定位 + 配置建议 | 康波理论 |

**标准化输出格式**:
```json
{
  "symbol": "600519",
  "analysis_type": "sentiment|factor|event|pattern|cycle",
  "framework": "value|trend",
  "scores": {
    "overall": 7.5,
    "dimensions": { ... }
  },
  "signals": ["买入信号", "趋势向上"],
  "confidence": 0.82,
  "reasoning": "...",
  "references": ["芒格检查清单第3项", "段永平三不原则"]
}
```

### 2.3 决策层（Decision Layer）— "手与刹车"

**核心目标**: 将研究层结论转化为具体交易动作，确保风险可控。

| 技能 | 输入 | 输出 | 知识融合 |
|------|------|------|---------|
| 风控守门员 | 持仓 + 评分 | 风险等级 + 预警 | 达利欧·全天候 + 利弗莫尔·止损 |
| 组合建议 | 多维评分 + 风控 | 买/卖/持有 + 仓位 | 段永平·集中 + 芒格·清单 |
| 研报生成器 | 全流程数据 | 结构化研报 | 全体系知识综合 |

**标准化输出格式**:
```json
{
  "symbol": "600519",
  "action": "买入|卖出|持有|观望",
  "position_pct": 15,
  "entry_price": 1850.00,
  "stop_loss": 1720.00,
  "target_price": 2100.00,
  "confidence": 0.78,
  "risk_level": "中",
  "framework_used": "value",
  "checklist_passed": true,
  "disclaimer": "仅供参考，不构成投资建议"
}
```

## 三、四智体角色定义

### 3.1 智能体分工

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Observer    │ ──→ │  Analyst    │ ──→ │ Strategist  │ ──→ │  Executor   │
│  观察智能体   │     │ 分析智能体   │     │ 决策智能体   │     │ 执行智能体   │
│             │     │             │     │             │     │             │
│ "我看到了    │     │ "数据显示    │     │ "综合判断    │     │ "风控通过    │
│  什么?"      │     │  什么?"     │     │  该怎么做?"  │     │  生成计划"   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
  调用数据层           调用研究层           调用决策层(主)       调用决策层(执行)
```

### 3.2 各智能体详细定义

| 属性 | Observer 观察者 | Analyst 分析师 | Strategist 策略师 | Executor 执行者 |
|------|----------------|---------------|-------------------|----------------|
| **核心职责** | 数据采集与监控 | 深度研究与推理 | 综合判断与拍板 | 风控落地与输出 |
| **调用技能** | market-data-fetcher, fundamentals-parser, alt-data-collector | sentiment-analyzer, factor-scoring-engine, technical-pattern-recognizer, kondratieff-cycle-analyzer, event-impact-analyzer | portfolio-suggestion, risk-guardrail(作为输入) | risk-guardrail(作为执行器), report-generator |
| **性格模型** | 敏锐、客观、不带情绪 | 理性、多维、芒格式思考 | 段永平式"本分"、达利欧式"极度透明" | 利弗莫尔式纪律、不打折执行 |
| **思维方式** | "事实是什么?" | "逻辑是什么?" | "应该怎么做?" | "如何安全执行?" |
| **输出物** | 标准化数据包 | 评分卡 + 信号 | 操作建议 | 交易计划 + 研报 |

### 3.3 协作流程

```
阶段1: Observer 感知
  │  "600519 今日放量上涨3.2%，北向资金净买入8亿"
  │  "白酒板块整体走强，政策面无重大变化"
  ↓
阶段2: Analyst 分析
  │  "因子评分: 价值8.2/成长6.5/质量8.8/动量7.1 → 综合7.7"
  │  "技术形态: 突破前高压力位，MACD金叉"
  │  "情绪: 市场情绪偏乐观(0.65)，但未达极端"
  ↓
阶段3: 偏差检测（强制）
  │  "认知偏差扫描: 未发现锚定效应/确认偏差"
  │  "芒格清单: 25项中22项通过"
  ↓
阶段4: Strategist 决策
  │  "综合评分7.7 > 阈值7.0，建议买入"
  │  "仓位建议: 15%（考虑当前市场环境）"
  ↓
阶段5: Executor 执行
  │  "风控检查: 通过（单笔风险3.2%，组合波动率12%）"
  │  "生成交易计划: 入场1850/止损1720/目标2100"
  │  "生成研报: 600519_深度分析_20260315.md"
  ↓
输出: 交易计划 + 研报 + 风险提示
```

## 四、数据流标准

### 4.1 层间数据传递协议

每层之间通过**标准化 JSON 格式**传递数据，确保模块解耦：

```
数据层输出 → 研究层输入:
  DataPacket {symbol, market, ohlcv[], financials{}, news[], timestamp}

研究层输出 → 决策层输入:
  AnalysisPacket {symbol, scores{}, signals[], patterns[], sentiment, cycle_position}

决策层输出 → 最终用户:
  DecisionPacket {action, position, entry/stop/target, risk_level, report}
```

### 4.2 框架选择逻辑

```
用户请求
  ├── --framework value  → 调用价值投资五步法
  │     └── 重点技能: fundamentals-parser, factor-scoring-engine
  │
  ├── --framework trend  → 调用趋势交易四象限
  │     └── 重点技能: technical-pattern-recognizer, sentiment-analyzer, kondratieff-cycle-analyzer
  │
  └── --framework both   → 双框架并行，交叉验证
        ├── 两套框架分别独立评估
        ├── 输出各自的评分卡/信号卡
        └── 综合建议考虑两套框架的共识与分歧
```

## 五、扩展性设计

### 5.1 新增技能

新技能只需遵循标准化输入/输出格式，放入对应层级的目录即可：
- 数据层: `skills/data-layer/`
- 研究层: `skills/research-layer/`
- 决策层: `skills/decision-layer/`

### 5.2 升级路径

```
当前: 路径A(AI工具) + 路径B(Python脚本) — 满足90%场景
  ↓
未来: 路径C(MCP Server) — 高频使用时升级
  ├── Finance MCP Server 封装所有数据接口
  ├── 支持数据缓存和增量更新
  └── 团队共享数据服务
```

### 5.3 新增知识源

在 `knowledge/` 目录添加新的知识提炼文件，然后在对应技能中引用即可。
